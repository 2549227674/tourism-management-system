## 1. 引言

### 1.1 编写目的

本文档用于总结旅游管理系统的开发过程、实际成果、进度执行情况、人员分工、测试结果、问题处理和经验教训，为课程设计验收和后续维护提供依据。

### 1.2 项目背景

旅游管理系统是软件工程课程设计项目，目标是通过模拟真实软件项目过程，完成从可行性研究、需求分析、软件设计、编码实现、测试验收到项目总结的完整流程。系统面向游客和管理员，提供景点、线路、订单、评论、收藏、公告和统计等功能。

## 5. 开发工作评价

### 5.1 对产品质量的评价

初版质量评价建议从以下方面填写：

1. 功能完整性：是否完成用户端和管理员端核心闭环；
2. 数据正确性：订单金额、线路名额、订单状态是否正确；
3. 安全性：普通用户是否被禁止访问管理员接口；
4. 可维护性：代码目录是否清晰，命名是否统一；
5. 文档一致性：交付文档与实际源程序是否一致。

### 5.2 对技术方法的评价

项目采用前后端分离架构。Spring Boot 适合实现 REST API 和业务逻辑，Vue 3 适合快速构建页面交互，MySQL 能够满足课程演示数据存储需求。该技术路线成熟、资料丰富，适合课程设计小组分工协作。

### 5.3 出错原因分析

可能出现的问题及原因示例：

| 问题 | 原因 | 解决方式 |
|---|---|---|
| 前后端字段不一致 | 接口文档更新不及时 | 以接口文档为准，联调后同步修改。 |
| 订单名额计算错误 | 取消和驳回未释放名额 | 增加订单状态流转测试用例。 |
| 管理员接口被普通用户访问 | 权限拦截规则不完整 | 增加后端角色校验。 |
| 数据库初始化失败 | 建表顺序或字符集设置错误 | 更新 `schema.sql` 和运行指南。 |

## 7. 经验与教训

### 7.1 经验

1. 先明确功能边界能有效避免项目过大；
2. 数据库和接口文档先行，有助于前后端并行开发；
3. 订单状态流转要尽早画清楚，否则容易出现数据不一致；
4. 文档应随代码更新，最后集中补文档容易遗漏。

### 7.2 教训

1. 如果没有统一接口字段，前后端联调会消耗大量时间；
2. 管理员和普通用户权限不能只靠前端控制，后端必须校验；
3. 课程设计项目不宜加入支付、地图、推荐等复杂功能；
4. 测试数据需要提前准备，否则演示时难以覆盖完整流程。
