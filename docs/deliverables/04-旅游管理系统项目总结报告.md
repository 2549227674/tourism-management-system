## 5. 开发工作评价

### 5.1 对产品质量的评价

1. 功能完整性：用户端和管理员端核心闭环已全部完成，涵盖用户注册登录、景点线路浏览、订单预订与处理、评论审核、收藏管理、公告管理和后台统计；
2. 数据正确性：订单金额由后端根据线路单价和人数自动计算，线路名额在订单创建、取消和驳回时由后端正确维护；
3. 安全性：密码使用 BCrypt 加密存储，管理员接口通过 Spring Security 角色校验，普通用户访问管理员接口返回 403；
4. 可维护性：代码按 controller、service、mapper、entity、dto、config、common 分层，命名统一，目录清晰；
5. 文档一致性：交付文档与源程序的接口路径、数据库字段、状态值、运行命令保持一致。

### 5.2 对技术方法的评价

项目采用前后端分离架构。Spring Boot 3 配合 MyBatis-Plus 实现 REST API 和数据访问，Spring Security + JWT 实现认证鉴权，Vue 3 + Vite + Element Plus 实现前端页面交互，MySQL 8.0 保存业务数据。该技术路线成熟、资料丰富，适合课程设计小组分工协作。前后端通过统一 JSON 响应格式和 Axios 封装进行通信，联调效率较高。

### 5.3 出错原因与解决方式

| 问题 | 原因 | 解决方式 |
|---|---|---|
| 前后端字段不一致 | 接口文档更新不及时 | 以接口文档为准，联调后同步修改。 |
| 订单名额计算错误 | 取消和驳回未释放名额 | 在订单状态流转中统一处理名额释放。 |
| 管理员接口被普通用户访问 | 权限拦截规则不完整 | 增加后端 @PreAuthorize 角色校验。 |
| 数据库初始化失败 | 建表顺序或字符集设置错误 | 更新 schema.sql 和运行指南。 |
| 评论表名与 MySQL 关键字冲突 | comment 是 MySQL 保留字 | 将表名改为 travel_comment。 |

## 7. 经验与教训

### 7.1 经验

1. 先明确功能边界能有效避免项目过大，本项目严格控制在景点、线路、订单、评论、收藏、公告和统计范围内；
2. 数据库和接口文档先行，有助于前后端并行开发，减少联调冲突；
3. 订单状态流转要尽早设计清楚，本项目在设计阶段就明确了 PENDING、CONFIRMED、CANCELLED、REJECTED、COMPLETED 五种状态及对应的名额变化规则；
4. 文档应随代码更新，最后集中补文档容易遗漏和不一致。

### 7.2 教训

1. 如果没有统一接口字段命名，前后端联调会消耗大量时间；
2. 管理员和普通用户权限不能只靠前端控制，后端必须做角色校验；
3. 课程设计项目不宜加入支付、地图、推荐等复杂功能，否则难以在课程周期内完成；
4. 测试数据需要提前准备，演示时需要覆盖从预订到完成的完整流程。

## 8. 结论

旅游管理系统围绕旅游信息管理和线路预订建立了完整的业务闭环，涵盖用户管理、景点管理、线路管理、订单管理、评论审核、收藏管理、公告管理和数据统计共 8 个功能模块。系统采用 Java 17 + Spring Boot 3 + MyBatis-Plus + Vue 3 + Element Plus + MySQL 8.0 技术栈，共实现 50 个 REST 接口、20+ 前端页面、8 张数据库表。

后端 `mvn clean test` 通过，前端 `npm run build` 构建成功，100 个前端功能测试用例全部通过，运行指南验收检查表 12 项全部通过。当前无阻塞验收的遗留缺陷。

本项目适合作为软件工程课程设计的参考案例，在技术选型、功能范围、文档规范和开发流程方面均符合课程要求。
