# 10-交付文档更新规则

## 1. 目标

本规则用于保证旅游管理系统最终提交的交付文档、开发文档和源代码保持一致，避免出现“文档写了但系统没有实现”或“代码实现了但文档没有说明”的问题。

## 2. 文档分类

### 2.1 最终交付文档

| 文档 | 更新时机 |
|---|---|
| 可行性研究报告 | 功能范围、成本收益、技术路线变化时。 |
| 需求分析规格说明书 | 新增、删除或修改功能需求时。 |
| 软件设计规格说明书 | 架构、模块、数据库、接口、页面变化时。 |
| 项目总结报告 | 项目完成后根据真实结果补充。 |
| 源程序说明与运行指南 | 运行命令、环境、账号、部署步骤变化时。 |

### 2.2 开发指导文档

| 文档 | 更新时机 |
|---|---|
| 技术栈与开发约束 | 技术版本或工具变化时。 |
| 模块规划 | 功能优先级变化时。 |
| 数据库设计说明 | 表、字段、索引、状态值变化时。 |
| 后端接口设计说明 | 接口路径、请求、响应、错误码变化时。 |
| 前端页面与交互说明 | 页面路由、交互流程变化时。 |
| 权限与业务规则说明 | 角色权限、状态流转、业务规则变化时。 |
| 测试用例与验收标准 | 修复缺陷、新增功能、验收口径变化时。 |

## 3. 更新触发条件

只要出现以下情况，必须同步更新文档：

1. 新增或删除功能模块；
2. 修改接口路径、请求字段或响应字段；
3. 修改数据库表名、字段名、字段类型或状态值；
4. 修改订单、评论、线路等状态流转规则；
5. 修改前端路由或页面操作流程；
6. 修改默认账号、运行端口或启动命令；
7. 修复影响验收的关键缺陷；
8. 最终验收前发现文档与代码不一致。

## 4. 文档和代码映射

| 代码变化 | 必须更新的文档 |
|---|---|
| 新增后端接口 | `04-后端接口设计说明.md`、软件设计规格说明书 |
| 修改数据库字段 | `03-数据库设计说明.md`、需求规格说明书、软件设计规格说明书 |
| 新增前端页面 | `05-前端页面与交互说明.md`、README |
| 修改权限规则 | `06-权限与业务规则说明.md`、需求规格说明书 |
| 修改订单状态 | `06-权限与业务规则说明.md`、`08-测试用例与验收标准.md` |
| 修改运行命令 | `05-源程序说明与运行指南.md`、README |
| 新增测试用例 | `08-测试用例与验收标准.md`、项目总结报告 |

## 5. 版本记录格式

每个重要文档底部可增加版本记录：

```markdown
## 版本记录

| 版本 | 日期 | 修改人 | 修改内容 |
|---|---|---|---|
| v0.1 | YYYY-MM-DD | 成员 D | 创建初版。 |
| v0.2 | YYYY-MM-DD | 成员 A | 更新订单接口和状态规则。 |
```

## 6. 最终提交前一致性检查

| 检查项 | 检查方法 | 负责人 |
|---|---|---|
| 技术栈一致 | README、CLAUDE.md、实际代码依赖对比 | 成员 A、D |
| 数据库一致 | SQL 脚本与数据库设计文档对比 | 成员 C |
| 接口一致 | 后端 Controller 与接口文档对比 | 成员 A |
| 页面一致 | 前端路由与页面说明对比 | 成员 B |
| 权限一致 | 实际登录角色测试与权限矩阵对比 | 成员 A、C |
| 测试一致 | 测试用例与项目总结对比 | 成员 C、D |
| 运行指南有效 | 按指南从零启动系统 | 全体 |

## 7. 项目总结报告更新要求

项目总结报告不能只保留模板，最终必须补充：

1. 实际完成的功能列表；
2. 未完成或简化的功能说明；
3. 实际进度和计划偏差；
4. 每位成员实际贡献；
5. 测试执行结果；
6. 主要问题和解决方法；
7. 经验与教训。

## 8. 提交包检查清单

最终压缩包建议包含：

```text
旅游管理系统-最终提交包/
├── backend/
├── frontend/
├── database/
├── docs/
├── README.md
├── CLAUDE.md
└── 项目演示说明或截图/
```

提交前确认：

- [ ] 所有 Markdown 文档可正常打开；
- [ ] 文件名与课程要求一致；
- [ ] 源程序没有缺少关键配置；
- [ ] 数据库脚本可以执行；
- [ ] 默认账号可登录；
- [ ] 演示流程能走通；
- [ ] 文档没有残留“图书管理系统”等无关内容；
- [ ] 项目总结报告已根据真实情况更新。
